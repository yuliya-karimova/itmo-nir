Детали реализации:

BDUI подход:
1. Создание condition блока через админку
2. Настройка заголовка (x-user-variant) и значения для сравнения
3. Настройка двух вариантов контента (trueBlock и falseBlock) через редактор
4. Сохранение страницы
5. Готово - без пересборки, без изменения кода

Classic подход:
1. Добавление endpoint на бэкенде (/api/banner-variant)
2. Изменение кода компонента HomePage.js
3. Пересборка фронтенда
4. Перезапуск сервисов
5. Деплой изменений

---

Оценка поддерживаемости:

BDUI:
- Изменения через админку (condition блок)
- Не требует пересборки фронтенда
- Не требует изменения кода
- Легко откатить (удалить или изменить condition блок в админке)
- Можно изменить условие или варианты контента без деплоя
- Оценка: 5/5 баллов

Classic:
- Требуется изменение кода компонента (HomePage.js)
- Требуется добавление endpoint на бэкенде
- Пересборка фронтенда обязательна
- Изменение вариантов требует изменения кода и пересборки
- Оценка: 2-3/5 баллов

---

Преимущества BDUI для A/B тестов:
- Быстрое создание и изменение экспериментов
- Не требует участия разработчиков для изменения контента
- Можно легко добавлять новые варианты или изменять условия
- Минимальное время доставки изменений в продакшн

Недостатки Classic для A/B тестов:
- Долгий цикл разработки (изменение кода → сборка → деплой)
- Требуется участие разработчиков для любых изменений
- Сложнее экспериментировать и итерировать
- Высокое время доставки изменений в продакшн
