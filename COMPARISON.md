# Методика сравнения BDUI и классического подхода

Этот документ описывает методику проведения сравнительного анализа двух подходов к разработке UI согласно НИР2.

## Структура проектов

- `bdui/` - Реализация Backend-Driven UI
- `classic/` - Реализация классического подхода

## Параметры сравнения

Согласно НИР2, сравнение проводится по 5 параметрам:

1. **Скорость доставки изменений (TTM)** - время от первого коммита до появления изменений у пользователей
2. **Объём передаваемых данных (трафик)** - размер ответов API и количество запросов
3. **Производительность и время загрузки UI** - метрики First Contentful Paint, Time to Interactive
4. **Использование ресурсов (CPU, RAM)** - нагрузка на клиент и сервер
5. **Простота поддержки и разработки** - субъективная оценка удобства работы

## Сценарии тестирования

### S1: Малая правка
**Описание**: Замена текста кнопки и цвета фона страницы

**BDUI**: 
- Изменить данные в админ-панели или через API
- Деплой на сервер
- Изменения сразу видны пользователям

**Classic**:
- Изменить код компонента страницы
- Пересобрать фронтенд
- Деплой новой версии
- Пользователи получат изменения после обновления

**Параметры**: TTM, трудоёмкость

### S2: Новая форма
**Описание**: Добавление формы из 5 полей с валидацией

**BDUI**:
- Добавить новый тип блока через контракты
- Реализовать компонент рендерера
- Настроить форму в админ-панели

**Classic**:
- Создать новый компонент формы
- Добавить валидацию
- Интегрировать в страницу
- Пересобрать и задеплоить

**Параметры**: Трафик, производительность, ресурсы

### S3: A/B-эксперимент
**Описание**: Показываем 50% пользователей альтернативный баннер

**BDUI**:
- Настроить логику на сервере для выбора варианта
- Изменить данные через админ-панель
- Изменения применяются мгновенно

**Classic**:
- Добавить логику выбора варианта в код
- Пересобрать и задеплоить
- Все пользователи получат новую версию одновременно

**Параметры**: TTM, поддерживаемость

### S4: Оффлайн-режим
**Описание**: Открытие уже посещённого экрана без сети

**BDUI**:
- Зависит от кэширования описания UI
- Может не работать без сети

**Classic**:
- UI уже в приложении
- Может работать с закэшированными данными

**Параметры**: Производительность, ресурсы

## Инструменты для измерений

### Скорость доставки изменений (TTM)
- GitLab CI / GitHub Actions - фиксация времени сборки/деплоя
- Jira / Issues - дата создания задачи
- Метрика: время от первого коммита до появления на тестовом сервере (минуты)

### Сетевой трафик
- Chrome DevTools (Network tab)
- Postman Runner - массовый прогон запросов
- Метрики:
  - Сумма переданных килобайт на один экран
  - Количество HTTP-запросов на один экран

### Производительность интерфейса
- Lighthouse CLI
- Chrome Web Vitals
- Метрики:
  - First Contentful Paint (мс)
  - Time to Interactive (мс)
  - Cumulative Layout Shift

### CPU/RAM
- pprof (Go) - нагрузка сервера
- Chrome Performance - нагрузка клиента
- top / htop - быстрая проверка
- Метрики:
  - Пик % CPU во время загрузки экрана
  - Пик RSS, МБ (оперативная память)

### Удобство поддержки
- Учёт трудозатрат в Jira (Work Log)
- Короткий опрос в Google Forms (шкала 1–5)
- Метрики:
  - Часы разработчика/задачу
  - Средний субъективный балл удобства

## Алгоритм проведения измерений

1. **Подготовка окружения**:
   - Разворачиваем оба проекта (BDUI и Classic)
   - Включаем одинаковый CI/CD pipeline
   - Настраиваем инструменты мониторинга

2. **Реализация сценария**:
   - Создаём issue в системе задач
   - Разработчик выполняет изменения
   - Pipeline фиксирует время начала и конца сборки

3. **Замер TTM**:
   - Экспортируем таймстемпы из CI/CD
   - Рассчитываем время доставки

4. **Нагрузочные и сетевые замеры**:
   - Запускаем Lighthouse 5 раз, берём медиану
   - Собираем HAR-файлы DevTools
   - Считаем суммарный объём и количество запросов

5. **Профилирование ресурсов**:
   - Для сервера - pprof
   - Для клиента - Chrome Performance snapshot
   - Фиксируем пик RAM

6. **Оценка поддерживаемости**:
   - Экспертная самооценка по 5 аспектам:
     - Ясность архитектурного решения
     - Объём повторяющегося кода
     - Сложность отладки
     - Наличие и полнота документации
     - Субъективная лёгкость "вхождения" в код
   - Каждый аспект оценивается по 5-балльной шкале
   - Итоговая оценка = среднее значение

7. **Нормализация в 5-балльную шкалу**:
   - Для количественных метрик используем линейное нормирование
   - Пороговые значения из критериев раздела 3 НИР2

8. **Сводная таблица и визуализация**:
   - Формируем матрицу критериев (2 × 5)
   - Строим диаграмму в Jupyter + Matplotlib

## Контроль достоверности

- Повторяем каждый сценарий не менее трёх раз, используем медиану
- Сбрасываем кэш браузера перед каждым прогоном
- Выделяем из CI/CD логов только успешные сборки
- Храним артефакты (HAR, pprof, отчёты Lighthouse) для верификации

## Порядок интерпретации результатов

1. Для каждого подхода суммируем баллы по пяти параметрам (максимум 25)
2. Анализируем, в каких сценариях наблюдается наибольший разрыв > 1 балл
3. Делаем выводы о применимости:
   - Если BDUI выигрывает ≥ 3 балла в S1–S3 и не проигрывает критично в S4 → подходит для продуктов с частыми изменениями
   - Если классика опережает по производительности и ресурсам ≥ 2 балла → остается предпочтительной для статичных или офлайн-критичных приложений

## Запуск проектов для сравнения

### BDUI
```bash
cd bdui
docker compose up --build
```
- Frontend: http://localhost:3000
- Backend: http://localhost:3001
- BFF: http://localhost:3002
- Admin: http://localhost:3003

### Classic
```bash
cd classic
docker compose up --build
```
- Frontend: http://localhost:3010
- Backend: http://localhost:3011

**Важно**: Проекты настроены на разные порты, поэтому можно запускать оба одновременно для сравнения.
