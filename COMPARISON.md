# Методика сравнения BDUI и классического подхода

Этот документ описывает методику проведения сравнительного анализа двух подходов к разработке UI согласно НИР2.

## Структура проектов

- `bdui/` - Реализация Backend-Driven UI
- `classic/` - Реализация классического подхода

## Параметры сравнения

Согласно НИР2, сравнение проводится по 5 параметрам:

1. **Скорость доставки изменений (TTM)** - время от начала внесения изменения до момента, когда обновлённый интерфейс становится доступен пользователю по развернутому адресу приложения
2. **Объём передаваемых данных (трафик)** - размер ответов API и количество запросов
3. **Производительность и время загрузки UI** - метрики First Contentful Paint, Time to Interactive
4. **Использование ресурсов (CPU, RAM)** - нагрузка на клиент и сервер
5. **Простота поддержки и разработки** - экспертная оценка трудоёмкости внесения изменений, количества затрагиваемых компонентов и необходимости пересборки приложения

## Сценарии тестирования

### S1: Простая правка
**Описание**: Изменить текст в баннере и добавить новый текстовый блок

**BDUI**: 
- Открыть админ-панель
- Найти блок баннера, изменить заголовок и подзаголовок
- Добавить новый текстовый блок через интерфейс
- Сохранить изменения
- Изменения становятся доступными без пересборки frontend-приложения (0-5 минут)

**Classic**:
- Открыть файл компонента страницы
- Изменить захардкоженные данные баннера в коде
- Добавить новый JSX блок в компонент
- Пересобрать фронтенд
- Деплой новой версии
- Требуется пересборка (10-30 минут)

**Параметры**: TTM, трудоёмкость

### S2: Сложная правка - Новая страница
**Описание**: Создать новую страницу "О нас" с баннером, текстовым блоком и блоком карточек

**BDUI**:
- Открыть админ-панель
- Создать новую страницу через интерфейс
- Добавить блоки (banner, text, cards) через визуальный редактор
- Заполнить данные для каждого блока
- Сохранить
- Страница доступна сразу (0-5 минут)

**Classic**:
- Создать новый компонент AboutPage.js
- Написать весь JSX с захардкоженными данными
- Добавить роут в App.js
- Создать CSS файл
- Пересобрать фронтенд
- Деплой новой версии
- Требуется создание компонента и пересборка (20-40 минут)

**Параметры**: TTM, трудоёмкость

### S3: Новая форма
**Описание**: Добавление формы из 5 полей с валидацией

**BDUI**:
- Использовать поддерживаемый тип блока формы.
- Настроить схему формы: поля, подписи, обязательность заполнения и правила базовой валидации.
- Разместить форму на странице через административную панель.
- Сохранить изменения.
- Проверить отображение формы во frontend-приложении.
- При наличии поддерживаемого типа блока изменение не требует пересборки frontend-приложения.

**Важное уточнение:**
Если тип блока формы отсутствует в рендерере, требуется разработка нового frontend-компонента, и преимущество BDUI по TTM уменьшается.

**Classic**:
- Создать или изменить компонент формы во frontend-коде.
- Добавить поля формы и правила валидации.
- Интегрировать форму в страницу.
- Выполнить сборку и повторное развёртывание frontend-приложения (20-40 минут).

**Параметры**: TTM, трафик, производительность, ресурсы

### S4: A/B-эксперимент
**Описание**: Показываем 50% пользователей альтернативный баннер

**BDUI**:
- Настроить логику на сервере для выбора варианта
- Изменить данные через админ-панель
- Изменения становятся видимыми после сохранения конфигурации и повторного запроса страницы клиентом (несущественно по времени)

**Classic**:
- Добавить логику выбора варианта в код
- Пересобрать и задеплоить
- После сборки и развёртывания новая логика становится доступна пользователям при загрузке обновлённой версии приложения

**Параметры**: TTM, поддерживаемость

### S5: Оффлайн-режим
**Описание**: Открытие уже посещённого экрана без сети

**BDUI**:
- Требует кэширования не только статических ресурсов, но и схемы интерфейса
- При наличии Service Worker и кэша ранее посещённая страница может быть доступна без сети

**Classic**:
- Структура интерфейса включена в клиентский бандл
- Для работы без сети требуется кэширование данных и статических ресурсов

**Параметры**: Производительность, ресурсы

## Инструменты для измерений

### Скорость доставки изменений (TTM)
- GitHub — фиксация изменений в репозитории
- Railway Deployments — фиксация времени сборки и развёртывания
- Логи сценариев — фиксация времени начала изменения и момента появления результата
- Метрика: время от начала внесения изменения до появления результата по развернутому адресу приложения

### Сетевой трафик
- Chrome DevTools (Network tab)
- Postman Runner - массовый прогон запросов
- Метрики:
  - Сумма переданных килобайт на один экран
  - Количество HTTP-запросов на один экран

### Производительность интерфейса
- Lighthouse CLI
- Chrome Web Vitals
- Метрики:
  - First Contentful Paint (мс)
  - Time to Interactive (мс)
  - Cumulative Layout Shift

### CPU/RAM
- Railway Metrics — оценка нагрузки серверных сервисов
- Chrome Performance — оценка нагрузки клиентской части
- top / htop — дополнительная проверка при локальном запуске

### Удобство поддержки
- Анализ состава действий в каждом сценарии
- Фиксация необходимости изменения кода, сборки и повторного развёртывания
- Экспертная оценка трудоёмкости внесения изменения

## Алгоритм проведения измерений

1. **Подготовка окружения**:
   - Разворачиваем оба проекта (BDUI и Classic)
   - Включаем одинаковый CI/CD pipeline
   - Настраиваем инструменты мониторинга

2. **Реализация сценария**:
   - Для каждого сценария фиксируются временные метки: начало изменения, завершение изменения, завершение сборки или развёртывания, появление результата в интерфейсе.
   - Для Classic дополнительно учитывается время сборки и развёртывания.
   - Для BDUI при изменениях через административную панель фиксируется время от начала настройки до появления результата во frontend-приложении.

3. **Замер TTM**:
   - Экспортируем таймстемпы из CI/CD
   - Рассчитываем время доставки

4. **Нагрузочные и сетевые замеры**:
   - Запускаем Lighthouse 5 раз, берём медиану
   - Собираем HAR-файлы DevTools
   - Считаем суммарный объём и количество запросов

5. **Профилирование ресурсов**:
   - Для сервера - pprof
   - Для клиента - Chrome Performance snapshot
   - Фиксируем пик RAM

6. **Оценка поддерживаемости**:
   - Экспертная самооценка по 5 аспектам:
     - Ясность архитектурного решения
     - Объём повторяющегося кода
     - Сложность отладки
     - Наличие и полнота документации
     - Субъективная лёгкость "вхождения" в код
   - Каждый аспект оценивается по 5-балльной шкале
   - Итоговая оценка = среднее значение

7. **Нормализация в 5-балльную шкалу**:
   - Для количественных метрик используем линейное нормирование
   - Пороговые значения из критериев раздела 3 НИР2

8. **Сводная таблица и визуализация**:
   - Формируем матрицу критериев (2 × 5)
   - Строим диаграмму в Jupyter + Matplotlib

## Контроль достоверности

- Повторяем каждый сценарий не менее трёх раз, используем медиану
- Сбрасываем кэш браузера перед каждым прогоном
- Выделяем из CI/CD логов только успешные сборки
- Храним артефакты (HAR, pprof, отчёты Lighthouse) для верификации

## Порядок интерпретации результатов

1. Для каждого подхода суммируем баллы по пяти параметрам (максимум 25)
2. Анализируем, в каких сценариях наблюдается наибольший разрыв > 1 балл
3. Делаем выводы о применимости:
   - Если BDUI показывает устойчивое преимущество по TTM в сценариях S1–S4 и не имеет критической деградации по производительности, ресурсам и офлайн-режиму, подход можно считать обоснованным для продуктов с частыми изменениями интерфейса.
   - Если Classic существенно превосходит BDUI по производительности, трафику или требованиям к офлайн-режиму, он может оставаться предпочтительным для статичных и производительно-критичных приложений.

## Запуск проектов для сравнения

### BDUI
```bash
cd bdui
docker compose up --build
```
- Frontend: http://localhost:3000
- Backend: http://localhost:3001
- BFF: http://localhost:3002
- Admin: http://localhost:3003

### Classic
```bash
cd classic
docker compose up --build
```
- Frontend: http://localhost:3010
- Backend: http://localhost:3011

**Важно**: Проекты настроены на разные порты, поэтому можно запускать оба одновременно для сравнения.
