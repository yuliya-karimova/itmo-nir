# Classic Frontend Approach Demo

Демонстрационное приложение для сравнения классического подхода к разработке UI с Backend-Driven UI.

## Архитектура

- **Backend** (порт 3011) - API для управления данными (только сырые данные, без структуры страниц)
- **Frontend** (порт 3010) - Публичная часть сайта с жестко закодированными компонентами страниц

## Ключевые отличия от BDUI

В классическом подходе:
- **Backend не знает про страницы** - отдает только данные по эндпоинтам
- **Frontend полностью контролирует структуру UI** - каждая страница сама определяет, какие эндпоинты вызывать
- **Структура страницы закодирована в компоненте** - порядок элементов, какие секции показывать
- **Для изменения структуры нужно изменить код фронтенда** и пересобрать приложение

## API Endpoints

Backend предоставляет только один тип данных:

- `GET /api/features` - Получить список фичей для главной страницы
- `GET /api/features/hello` - Получить список фичей для страницы hello

**Все остальные данные (баннеры, тексты, кнопки) захардкожены на фронтенде!**

Backend **не знает**:
- Какие страницы существуют
- Какая структура у страниц
- Какие данные нужны для каждой страницы (кроме features)

## Запуск

```bash
docker compose up --build
```

После запуска:
- Публичный сайт: http://localhost:3010
- Backend API: http://localhost:3011

## Структура проекта

```
classic/
├── backend/          # Backend API (Node.js + Express) - только данные
│   └── data/
│       └── data.json # Сырые данные без структуры страниц
├── frontend/         # Публичный фронтенд (React)
│   └── src/
│       └── pages/    # Жестко закодированные компоненты страниц
│           ├── HomePage.js   # Сама определяет, какие эндпоинты вызывать
│           ├── InfoPage.js   # Сама определяет, какие эндпоинты вызывать
│           └── HelloPage.js  # Сама определяет, какие эндпоинты вызывать
└── docker-compose.yml
```

## Особенности

- **Классический подход**: Структура UI полностью закодирована во фронтенде
- **Только данные**: Backend отдает только сырые данные, без описания структуры
- **Статическая структура**: Каждая страница имеет свой компонент с предопределенной структурой
- **Контроль на фронтенде**: Фронтенд сам решает, какие эндпоинты вызывать и как отображать данные
- **Изменения требуют пересборки**: Любое изменение структуры страницы требует изменения кода и пересборки

## Пример работы

### HomePage
```javascript
// Почти все данные захардкожены на фронтенде
const banner = {
  title: 'Добро пожаловать!',
  subtitle: 'Classic Frontend Approach Demo',
  // ...
};

const about = {
  title: 'О проекте',
  content: '...'
};

const buttons = [
  { id: '1', text: 'info', link: '/info', style: 'primary' }
];

// Только features получаем с бэка
const [features] = await axios.get('/api/features');

// Структура страницы закодирована здесь
return (
  <div>
    <Banner data={banner} />      {/* Захардкожено */}
    <Buttons data={buttons} />    {/* Захардкожено */}
    <About data={about} />        {/* Захардкожено */}
    <Features data={features} />  {/* С бэка */}
  </div>
);
```

### Изменение структуры
Чтобы изменить порядок элементов или добавить новую секцию:
1. Открыть `frontend/src/pages/HomePage.js`
2. Изменить код компонента (данные захардкожены прямо в компоненте)
3. Пересобрать: `docker compose up -d --build frontend`

Backend при этом не меняется! Backend используется только для получения features.

### Изменение данных
- **Баннеры, тексты, кнопки**: Изменить прямо в коде компонента страницы
- **Features**: Изменить в `backend/data/data.json` или через API `PUT /api/features`
