# Изменения в классическом подходе

## Переделка архитектуры

### Что изменилось:

1. **Backend больше не знает про страницы**
   - Удален файл `pages.json`
   - Удалены эндпоинты `/api/pages`, `/api/pages/:slug`
   - Backend теперь отдает только features по эндпоинтам

2. **Минимальные эндпоинты API:**
   - `GET /api/features` - список фичей для главной страницы
   - `GET /api/features/hello` - список фичей для страницы hello
   - Все остальные данные (баннеры, тексты, кнопки) захардкожены на фронтенде

3. **Frontend полностью контролирует структуру:**
   - Почти все данные захардкожены прямо в компонентах страниц
   - Только features получаются с бэкенда
   - Структура страницы закодирована в компоненте
   - Порядок элементов определяется во фронтенде

### Пример:

**Раньше:**
```javascript
// Backend отдавал структурированные данные для страницы
GET /api/pages/home → {
  data: {
    banner: {...},
    aboutSection: {...},
    features: {...}
  }
}
```

**Теперь:**
```javascript
// Почти все данные захардкожены на фронтенде
const banner = {
  title: 'Добро пожаловать!',
  subtitle: 'Classic Frontend Approach Demo',
  // ...
};

const about = {
  title: 'О проекте',
  content: '...'
};

// Только features получаем с бэка
const [features] = await axios.get('/api/features');

// Структура страницы закодирована в компоненте
return (
  <div>
    <Banner data={banner} />      {/* Захардкожено */}
    <About data={about} />          {/* Захардкожено */}
    <Features data={features} />    {/* С бэка */}
  </div>
);
```

### Преимущества:

- ✅ Backend не знает про страницы - только данные
- ✅ Frontend полностью контролирует структуру UI
- ✅ Более честное сравнение с BDUI
- ✅ Изменение структуры требует изменения кода фронтенда (классический подход)
