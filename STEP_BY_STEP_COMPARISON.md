# Пошаговая инструкция по сравнению BDUI и классического подхода

Детальное руководство для проведения практического сравнения согласно методике НИР2.

## Варианты проведения сравнения TTM

### Вариант A: Локальное сравнение (быстро, для тестирования)

Сравнение на локальных машинах. Подходит для быстрой проверки, но не учитывает:
- Время сборки на удаленном сервере
- Время деплоя
- Распространение через CDN

**Используйте этот вариант для:**
- Первичной проверки функциональности
- Отладки процесса измерений
- Быстрых итераций

### Вариант B: Облачное развертывание (рекомендуется для финального сравнения) ⭐

Развертывание на облачных платформах для более реалистичного сравнения.

**Преимущества:**
- Учитывает реальное время сборки и деплоя
- Показывает разницу в скорости доставки изменений
- Более точные результаты для НИР3

**Рекомендуемые платформы:**
- **Railway** (лучший выбор) - бесплатный тариф, поддержка Docker
- **Vercel** - для фронтенда, быстрый CDN
- **Render** - альтернатива Railway

**Подробные инструкции:** См. [DEPLOYMENT_GUIDE.md](DEPLOYMENT_GUIDE.md)

**Быстрый старт с Railway:**

1. Создайте аккаунт на https://railway.app
2. Создайте новый проект из GitHub репозитория
3. Для BDUI: добавьте сервисы (backend, bff, frontend, admin)
4. Для Classic: добавьте сервисы (backend, frontend)
5. Настройте переменные окружения
6. Railway автоматически задеплоит при каждом push

**Измерение TTM на облаке:**

```bash
# Фиксируем время коммита
echo "Commit: $(date -u +"%Y-%m-%dT%H:%M:%SZ")" > cloud_ttm.txt

# Делаем изменения и пушим
git add .
git commit -m "TTM test: Change banner"
git push

# Railway/Vercel начнут автоматический деплой
# Время деплоя видно в Dashboard

# Проверяем доступность изменений
curl https://your-app.railway.app
echo "Visible: $(date -u +"%Y-%m-%dT%H:%M:%SZ")" >> cloud_ttm.txt
```

#### Шаг 2.4: Измерение ресурсов

**CPU и RAM на клиенте:**

**Перед добавлением формы:**

1. Откройте страницу без формы
2. Откройте Chrome DevTools (F12)
3. Перейдите на вкладку **Performance**
4. Нажмите **Record** (круглая кнопка вверху слева)
5. Обновите страницу (F5 или Ctrl+R)
6. Дождитесь полной загрузки страницы
7. Остановите запись (нажмите Stop или кнопку Record снова)

**Как посмотреть пиковые значения CPU и RAM:**

После остановки записи вы увидите график с временной шкалой:

1. **График JS Heap (память) - самый простой способ:**
   - В верхней части Performance есть переключатель метрик (легенда с цветными полосками)
   - Найдите "JS heap" (синяя полоска) и убедитесь, что она включена (галочка)
   - На графике вы увидите синюю линию - это использование памяти JavaScript
   - В легенде вверху указан диапазон: например, "JS heap [1.3 MB – 5.9 MB]"
   - **Пиковое значение RAM** = максимальное значение из диапазона (в примере: 5.9 MB)
   - Или наведите курсор на самую высокую точку синей линии - внизу в Summary увидите текущее значение
   - В Summary внизу экрана также показано текущее значение: "JS heap: 5.4 MB"

2. **Другие метрики памяти (опционально):**
   - "Nodes" (зеленая полоска) - количество DOM узлов
   - "Documents" (красная полоска) - количество документов
   - "Listeners" (золотая полоска) - количество обработчиков событий
   - Эти метрики показывают структуру страницы, но не объем памяти напрямую

3. **Для измерения CPU:**
   - **Способ 1 - Использовать Total Blocking Time (TBT) из Lighthouse:**
     - TBT уже измерен в Шаге 2.3
     - Это метрика, которая отражает нагрузку на CPU
     - Запишите TBT из Lighthouse отчета
   
   - **Способ 2 - Посмотреть в Summary панели Performance:**
     - Внизу экрана Performance есть панель "Summary" или "Bottom-Up"
     - Найдите "Total Time" или "Self Time" для главного потока
     - Это время выполнения JavaScript, которое косвенно показывает CPU нагрузку
   
   - **Способ 3 - Использовать метрики сервера (Railway):**
     - См. раздел "CPU и RAM на сервере" ниже
     - Метрики сервера показывают использование CPU nginx/Node.js сервера

**Рекомендация:**
- Для **RAM**: используйте максимальное значение JS heap из легенды или графика
- Для **CPU**: используйте TBT из Lighthouse (уже измерено) или метрики сервера Railway

4. Экспортируйте профиль: Right-click на графике → "Save profile"
5. Сохраните как `bdui_s2_before_performance.json` в `comparison_results/artifacts/profiles/`

## Сценарий S3: A/B-эксперимент (TTM, поддерживаемость)

### BDUI подход

#### Шаг 3.1: Создание condition блока через админку

1. Откройте админ-панель BDUI (например, `http://localhost:3003`)
2. Откройте нужную страницу для редактирования
3. Добавьте блок типа **"Условный блок"** (condition)
4. Заполните поля:
   - **Имя заголовка**: `x-user-variant` (или любой другой заголовок, который будет передаваться в запросе)
   - **Значение для сравнения**: `A` (для варианта A) или `B` (для варианта B)
   - **Блок если условие true**: 
     - Выберите **Тип блока** из выпадающего списка (например, `banner`, `text`, `cards`)
     - После выбора типа откроется редактор с полями этого типа блока
     - Заполните все необходимые поля (например, для banner: title, subtitle, imageUrl, buttonText, buttonLink)
   - **Блок если условие false**:
     - Выберите **Тип блока** из выпадающего списка
     - Заполните поля для этого блока (может быть тот же тип или другой)
5. Сохраните страницу

**Преимущества:**
- Можно использовать любой существующий тип блока (banner, text, cards, form и т.д.)
- Редактирование через удобный интерфейс на основе контракта, без необходимости писать JSON вручную
- Все поля валидируются автоматически

**Как это работает:**
- BFF проверяет заголовок запроса (например, `x-user-variant`)
- Если заголовок равен значению из поля "Значение для сравнения" - показывается блок из "Блок если условие true"
- Иначе показывается блок из "Блок если условие false"
- Condition блок обрабатывается на сервере, фронтенд получает уже готовый блок (trueBlock или falseBlock)

```javascript
// Кэшируем последнее описание страницы
self.addEventListener('fetch', (event) => {
  if (event.request.url.includes('/api/page/')) {
    event.respondWith(
      caches.match(event.request).then((response) => {
        return response || fetch(event.request).then((response) => {
          const responseClone = response.clone();
          caches.open('bdui-cache').then((cache) => {
            cache.put(event.request, responseClone);
          });
          return response;
        });
      })
    );
  }
});
```
